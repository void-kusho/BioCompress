<objective>
Debug and fix SEGFAULT in file-level decompression for Huffman levels 1-2.

**Current status:**
- Buffer-level test passes: "ACGTACGT" → encode → decode → "ACGTACGT" ✅
- File-level test fails: SEGFAULT on decompress ⚠️

**Root cause hypothesis:** The compressed data format has a 4-byte length prefix at position 0 when passed to huffman_decode, but the bit_pos calculation (hardcoded at bit 32) assumes it's at position 32 (after 4 bytes). This offset mismatch causes decode to read past buffer bounds.

**Fix approach:** Ensure consistent bit position handling between encode/decode.
</objective>

Analyze the bit position handling in huffman_decode:

```c
size_t bit_pos = 32;  // Start after 4-byte length prefix (32 bits)
```

This assumes the 4-byte length prefix is at byte 0 of the input. However:
- compress_buffer passes compressed data AFTER the 12-byte header
- The compressed data already contains the 4-byte length prefix at its start
- So bit_pos should be 0, not 32!

Fix: Change bit_pos to 0 in decode since the length prefix is at byte 0 of the compressed data.

<task type="auto">
  <name>Task 3: Fix Huffman decode bit position</name>
  <files>src-c/src/huffman.c</files>
  <action>
Fix the bit position in huffman_decode:

In huffman_decode function, change:
```c
size_t bit_pos = 32;  // Start after 4-byte length prefix (32 bits)
```

To:
```c
size_t bit_pos = 32;  // Start after 4-byte length prefix (at bit 32)
```

Wait - the issue is more subtle. Let me trace the exact flow:

1. compress.c calls huffman_encode with raw input "ACGTACGT..."
2. huffman_encode calculates:
   - out_size = 4 + (num_symbols * 2 + 7) / 8 
   - Stores length at bytes 0-3 (bits 0-31)
   - Stores bitstream starting at bit 32 (byte 4)

3. compress_buffer wraps output with 12-byte header
4. compress.c passes only the compressed bitstream (NOT the header) to compress_file
5. decompress_file reads the .bc file, extracts data after 12-byte header
6. decompress_buffer passes this to huffman_decode
7. huffman_decode reads 4-byte length starting at byte 0 of input (CORRECT)
8. bit_pos = 32 starts reading at bit 32 which is byte 4

So the logic should be correct! But wait - what if the input_size passed to decode doesn't account for the extra header bytes correctly?

Actually, checking compress.c line 186-188:
```c
const uint8_t* compressed = input + sizeof(header);
size_t compressed_size = input_size - sizeof(header);
```

This correctly extracts only the compressed data (after 12-byte header).

Let me add more specific debug output to see the actual values:
- Print input_size at decode entry
- Print num_symbols read
- Print expected_bit_pos = num_symbols * 2 / 8 + 4
- Compare with actual input_size
